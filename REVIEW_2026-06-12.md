# Council Review — the-council (2026-06-12)

**Target:** github.com/adnanraf1q/the-council @ cbbc18a (self-review, first session of the council)
**Stated goal:** portable, machine-agnostic Claude Code skills; one shareable source of truth across machines, accounts, and teams.

## Executive summary

**Verdict: invest** (trivially — it's a day old, cheap to maintain, and already deployed to one machine). The repo does what it says: the skill is verified machine-agnostic and the install paths are real. The gaps are all about *sharing* — the thing the repo exists for: no license (so "shareable" is legally false), no staleness signal for copied deployments, and a security posture that hasn't been thought through for the symlink-auto-update convention it recommends.

### Top 5 actions (impact ÷ effort)

| # | Action | Effort | Why |
|---|---|---|---|
| 1 | Add a LICENSE (MIT) | quick-win | Public repo with no license = all-rights-reserved. Colleagues copying `council/` into work repos currently have no legal right to. One file fixes the repo's core promise. |
| 2 | Add a multi-project guard to SKILL.md | quick-win | Discovered live in session 1: invoked from a directory of 15 projects, the skill has no instruction to ask which app — it assumes cwd is the app. One line: "If the working directory contains multiple projects, ask which one before reading anything." |
| 3 | Declare SKILL.md canonical | quick-win | The 8-role panel is defined twice — README catalog table and SKILL.md. They will drift. One sentence in README ("summary only; SKILL.md is canonical") ends the race. |
| 4 | Pin-by-commit advice for work machines | quick-win | The symlink + auto-update convention means a compromised GitHub account pushes instructions that execute with your permissions on every linked machine. For work: clone at a pinned commit, review diffs before bumping. One README paragraph. |
| 5 | Frontmatter lint CI (name == folder, parseable YAML) | medium | A malformed SKILL.md doesn't error — the skill silently never appears. Ten-line GitHub Action catches it at push time. |

## Engineering panel

### Architect
1. **(med / quick-win)** Panel definition duplicated: `README.md:64–73` (catalog table) vs `council/SKILL.md:21–58`. Two sources of truth for the same 8 roles — first edit that touches only one starts the drift the repo's own Conventions section warns about. Declare SKILL.md canonical.
2. **(low / quick-win)** `README.md:96–109` ("Adding a new skill") requires manually syncing the catalog table — same drift class as #1, pre-installed for every future skill.
3. **(low / —)** `the-council/council/` stutters in paths and in `ln -s` targets (`README.md:42`). Cosmetic; not worth a rename now that it's pushed, but worth knowing it was noticed.

### Security reviewer
1. **(med / quick-win)** Supply-chain exposure by design: `README.md:48` celebrates that `git pull` updates every project on the machine at once. Skills are executed instructions — anyone who can push to this public repo's main (today: one GitHub account, no branch protection) can change what Claude Code does with the operator's permissions on every linked machine at next pull. Likelihood low, blast radius high. Mitigation for work machines: pin a commit, review diffs before bumping (→ top action #4). No other findings — repo is markdown only, no secrets, verified by grep audit this session.

### Performance, infra & scale
Skipped — no runtime, no data path. The one scale-relevant property is already right: the skill runs inline (no subagent fan-out) and SKILL.md is 64 lines, so its per-session context cost is negligible.

### Reliability (operator + QA)
1. **(med / quick-win)** Copied deployments (Options A and C — the work path) have **no staleness signal**: no version in frontmatter, no CHANGELOG. When the master improves, a copied `council/` in a work repo looks identical to current. A one-line `version:` bump convention, or a CHANGELOG section in README, makes drift visible.
2. **(low–med / medium)** The ONE gate worth adding: a frontmatter lint (GitHub Action, ~10 lines) asserting each skill folder's `SKILL.md` has parseable YAML and `name:` == folder name. The failure it catches is silent and nasty: malformed frontmatter doesn't error — the skill just never appears in Claude Code, and the operator debugs the wrong thing.
3. **(low / —)** `README.md:46` Windows `xcopy` fallback is untested on a real Windows box. Flag, don't fix.

## Product panel

### Product & market *(merged)*
1. **(high / quick-win)** Smallest step to real feedback is NOT more polish here — it's deploying `/council` into the work repo and running it on the actual app. This self-review is a smoke test, not market feedback. Everything above except the LICENSE can wait until after that first real run.
2. **(med / —)** Moat: none, and that's fine — state it plainly. It's a curated prompt; clonable in minutes. The durable value is the iteration loop (run → notice a miss → improve the master → push), which is exactly what the Conventions section already encodes.
3. **(med / —)** What NOT to build: a second skill before one is wanted twice, plugin/marketplace packaging, per-language variants. The catalog scaffolding already anticipates growth; let demand fill it.
4. **(low / quick-win)** Competitor check: Claude Code ships `/code-review` and `/security-review`, which overlap the engineering panel's bug-hunting. Council's differentiation is the product panel + verdict + ranked actions. One README sentence on "when to use council vs built-in /code-review" would prevent the obvious work-colleague question.

### Experience *(merged)*
1. **(med / quick-win)** First-run friction, observed live: invoked from a multi-project directory, the skill silently assumes cwd is the target. Session 1 had to improvise the "which app?" question. Encode it (→ top action #2).
2. **(low / —)** The report lands in the target repo's root with no guidance on whether to commit it. Teams will wonder. A line in SKILL.md's output section ("commit or gitignore per team preference") settles it.
3. **(low / —)** What works: the Critic is the memorable seat — it's the feature people will quote to each other, and the reason the name lands. Don't dilute it with caps or politeness in future edits.

### Compliance reviewer
1. **(high / quick-win)** **No LICENSE file.** Public repo, default all-rights-reserved. Every install path in the README invites copying into other codebases — including an employer's — with no grant to do so. MIT matches intent (→ top action #1).
2. **(low / —)** Reverse-flow IP risk — largely closed by operator setup: personal GitHub is never used on the work laptop, so no git path exists from work to this public repo. Residual is the human path only (manually re-typing a work-evolved improvement into the public master from home); relevant solely if the improvement encodes employer specifics. Awareness, not a convention. *Downgraded from med after review discussion.*

### The Critic (no filter)
A repository containing one 64-line markdown file now has a themed name chosen from a shortlist, a backstory, a README three times longer than the product, a memory entry, and a ceremonial self-review — the infrastructure-to-content ratio is exactly backwards, and the council is currently a government with no citizens. None of today's effort is wasted *if* Monday's plan actually happens: clone it at work, point it at the real app, and let it find one finding that changes what gets built that week. That is the only review of the-council that counts, and it hasn't happened yet. Until a second skill exists, "one source of truth across machines" is an elaborate way to say "a gist with a symlink" — which is fine, as long as nobody here mistakes naming things for shipping things.
